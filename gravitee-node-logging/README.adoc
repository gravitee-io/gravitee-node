= Gravitee Node Logging

== Description

The *Gravitee Node Logging* module provides a specialized logging infrastructure for Gravitee Node components.
It introduces the `NodeAwareLogger`, a decorator for SLF4J loggers that automatically enriches the Mapped Diagnostic Context (MDC) with node-specific information for every log call.

== NodeAwareLogger

The `NodeAwareLogger` ensures that important contextual information is always available in your logs, which is crucial for observability in distributed environments.
By default, it enriches the MDC with the following keys:

* `nodeId`: The unique identifier of the Gravitee Node.
* `nodeHostname`: The hostname where the node is running.
* `nodeApplication`: The name of the application.

=== Extensibility

The `NodeAwareLogger` is designed to be extensible. You can create your own specialized loggers by extending `NodeAwareLogger` and overriding the following methods:

* `registerLogSources(Map<Class<?>, Object> logSources)`: To provide additional context objects (e.g., an Execution Context, an HTTP Request, etc.).
* `registerLogEntries(Set<LogEntry<?>> registeredEntries)`: To define new MDC keys and how their values should be extracted from the registered log sources.

The `LogEntryFactory` provides utility methods to create `LogEntry` instances:
* `cached`: The value is extracted once and then cached.
* `refreshable`: The value is re-extracted at each log call.
* `fixed`: A constant value.

== NodeLoggerFactory

The `NodeLoggerFactory` is the entry point for obtaining `NodeAwareLogger` instances.
It acts as a singleton factory that manages the `Node` context.

=== Initialization

To benefit from node-specific enrichment, the factory must be initialized with a `Node` instance or a `Supplier<Node>`.
This is typically done during the node startup phase:

[source,java]
----
NodeLoggerFactory.init(node);
----

=== Obtaining a Logger

Once initialized (or even before, as it handles lazy initialization), you can obtain a logger just like you would with SLF4J's `LoggerFactory`:

[source,java]
----
private static final Logger logger = NodeLoggerFactory.getLogger(MyClass.class);
----

The returned logger will be a `NodeAwareLogger` instance wrapping the standard SLF4J logger.
If the `Node` context is not yet available at the time of the log call, it will gracefully fallback to standard logging without enrichment.

== Logging Configuration Override

The logging system can be configured via `gravitee.yml` (or environment variables) to control MDC filtering and log pattern formatting.
This is especially useful for Kubernetes/Helm deployments where modifying `logback.xml` is impractical.

Configuration is loaded by `LoggingOverrideConfiguration`, a Spring bootstrap bean registered in `SpringBasedContainer`.

=== MDC Configuration

Control which MDC keys are populated and how they are formatted by the `%mdcList` conversion word.

[source,yaml]
----
node:
  logging:
    mdc:
      include:
        - nodeId
        - nodeHostname
      format: "{key}: {value}"      # Format pattern for each entry (default: {key}: {value})
      separator: " | "              # Separator between entries (default: single space)
      nullValue: "-"                # Value for missing MDC entries (default: empty string)
----

When `include` is set, only the listed keys are populated into the MDC. Keys not in the list are never added, which affects all appenders (pattern-based and JSON).
When `include` is empty or absent, all MDC keys are populated (backward-compatible behavior).

The order of the `include` list is preserved and determines the rendering order of the `%mdcList` conversion word.

==== The `%mdcList` conversion word

A custom logback conversion word that renders all included MDC entries as a single formatted string.
It is registered programmatically at startup by `LoggingOverrideConfiguration`.

Example output with the configuration above:

----
nodeId: gw-1 | nodeHostname: host-alpha
----

Individual MDC values remain accessible via the standard `%X{key}` pattern.

[IMPORTANT]
.Using `%mdcList` directly in `logback.xml`
====
The programmatic registration happens *after* logback parses `logback.xml`.
If you want to use `%mdcList` directly in your `logback.xml` patterns, you must register the converter explicitly:

[source,xml]
----
<configuration>
    <conversionRule conversionWord="mdcList"
                    converterClass="io.gravitee.node.container.logback.MdcListConverter" />

    <!-- Now %mdcList can be used in any pattern -->
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>%d{HH:mm:ss.SSS} [%thread] [%mdcList] %-5level %logger{36} - %msg%n</pattern>
        </encoder>
    </appender>
</configuration>
----

This is not needed when using the pattern override via `gravitee.yml`, as the converter is already registered before the pattern is applied.
====

=== Pattern Override

Override logback console and file appender patterns from `gravitee.yml`:

[source,yaml]
----
node:
  logging:
    pattern:
      overrideLogbackXml: true
      console: "%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %mdcList %msg%n"
      file: "%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %mdcList %msg%n"
----

Pattern overrides only apply when `overrideLogbackXml` is `true`.
Only appenders using `PatternLayoutEncoder` are affected — JSON encoders and other encoder types are left untouched.

=== Environment Variable Overrides

All properties can be overridden via environment variables, following Gravitee's standard naming convention:

[cols="1,1"]
|===
| YAML property | Environment variable

| `node.logging.mdc.include[0]` | `GRAVITEE_NODE_LOGGING_MDC_INCLUDE_0`
| `node.logging.mdc.format` | `GRAVITEE_NODE_LOGGING_MDC_FORMAT`
| `node.logging.mdc.separator` | `GRAVITEE_NODE_LOGGING_MDC_SEPARATOR`
| `node.logging.mdc.nullValue` | `GRAVITEE_NODE_LOGGING_MDC_NULLVALUE`
| `node.logging.pattern.overrideLogbackXml` | `GRAVITEE_NODE_LOGGING_PATTERN_OVERRIDELOGBACKXML`
| `node.logging.pattern.console` | `GRAVITEE_NODE_LOGGING_PATTERN_CONSOLE`
| `node.logging.pattern.file` | `GRAVITEE_NODE_LOGGING_PATTERN_FILE`
|===

[TIP]
.Lombok integration
====
You can use Lombok's `@CustomLog` annotation to automatically inject this logger.
The project is configured via the `lombok.config` file at the root of the repository:

[source,properties]
----
lombok.log.custom.declaration = org.slf4j.Logger io.gravitee.node.logging.NodeLoggerFactory.getLogger(TYPE)
----

Then, simply use the annotation in your class:

[source,java]
----
@CustomLog
public class MyClass {
    public void myMethod() {
        log.info("Hello world");
    }
}
----
====
